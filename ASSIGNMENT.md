一、作业背景
我们是一家自研大模型产品公司，正在构建面向复杂业务场景的 Agent 框架。真实项目中，客户常提供扫描版 PDF、合同、产品手册、标准文件、金融材料、合规文档等非结构化资料。系统不仅要“能跑”，还要能解释、能测试、能定位问题、能适配不同业务。
本次作业以附件中的扫描版 PDF 为输入，模拟“文档理解 + 检索问答 + 可靠性验证”的真实交付场景。
附件：
- `GBT 1568-2008 键 技术条件.pdf`
该 PDF 是扫描件，通常没有可直接复制的文本层，包含中文正文、编号条款和表格内容。请不要手工录入全文来绕过解析过程。
二、作业目标
请设计并实现一个最小可运行的“智能文档问答 Agent”原型。
系统应能围绕附件 PDF 完成以下闭环：
1. 判断 PDF 类型并选择解析策略。
2. 提取或识别 PDF 中的正文、条款编号、表格信息。
3. 构建可检索知识库。
4. 接收用户问题，检索相关证据。
5. 生成答案，并返回来源页码/片段。
6. 对答案做基本自检：是否有依据、是否可能幻觉、是否需要拒答。
7. 给出测试方法和不同业务场景下的保障方案。
三、核心任务
请按照“作业目标”自行完成核心任务，交付一个可运行、可说明、可验证的智能文档问答 Agent 原型。
请你根据自己的理解，判断这个系统应该包含哪些模块、如何设计、如何实现、如何验证、如何应对边界情况，并在提交材料中说明你的思路、取舍和完成情况。
你可以自由选择技术路线和交互形式，但需要让我们能够复现、体验并评估你的方案。
四、提交要求
（1）GITHUB代码库（合理安排代码库结构）
（2）相关README文件
（3）演示材料：
请提供一段 5-10 分钟演示视频或清晰截图，展示：
1. 从部署/启动开始的完整流程。
2. PDF 解析结果，包括正文和表格。
3. 至少 5 个问题的问答结果，其中至少包含 1 个表格问题和 1 个无答案问题。
4. 来源引用和自检结果。
5. 测试或评估脚本运行结果。
五、我们会重点关注
本题质量优先，代码实现只是其中一部分。即使功能没有完全做完，只要思路扎实、取舍清楚、测试充分，也可能进入下一轮；反之，如果只交一个看似能跑但无法解释、无法测试、无法扩展的 Demo，会降低评价。
我们会重点看：
1. 你是否能识别扫描 PDF、表格、OCR、检索、生成、自检等关键问题。
2. 你是否能设计合理的 Agent / RAG 流程，而不是简单套壳。
3. 你是否能设计有效测试，覆盖正文、表格、无答案、模糊问题、OCR 错误和回归风险。
4. 你是否能说明方案如何迁移到金融、合规、客户交付等不同业务场景。
5. 你是否善用 AI / Agent 工具提升效率，并能校验 AI 输出、对结果负责。
6. 你是否有清晰的工程交付习惯，包括异常处理、配置、日志、可复现启动和文档说明。
